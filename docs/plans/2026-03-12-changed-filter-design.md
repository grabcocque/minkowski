# Changed<T> Filtering for Query Executor

## Problem

The query executor (`QueryPlanResult::for_each`, `for_each_raw`, `execute`) iterates
all matching archetypes unconditionally. The existing `world.query()` path supports
`Changed<T>` filtering that skips entire archetypes whose columns haven't been modified
since the last read. The query executor lacks this capability.

## Goal

Add `Changed<T>` support to the query executor consistent with the existing query path.
A plan built from `scan::<(Changed<Pos>, &Vel)>()` should skip archetypes where the
`Pos` column hasn't changed, using the same archetype-granularity tick comparison.

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Tick state location | Per-plan-instance `AtomicU64` on `QueryPlanResult` | Independent of `world.query()` tick state. Two plan instances for the same query type track changes independently. Consistent with QueryWriter's per-reducer `Arc<AtomicU64>` pattern. |
| Changed component extraction | New `WorldQuery::changed_ids()` trait method returning `FixedBitSet` | Extracts component IDs as static data while `Q` is in scope at `scan::<Q>()` time. No trait-object overhead at runtime. Consistent with existing `required_ids`/`mutable_ids` pattern. |
| Tick threading | Closure parameter `Tick` | Simpler than shared atomic. Tick loaded once by caller, passed through. No atomic loads in the hot loop. |
| `for_each` tick advancement | Immediate after scan completes | `for_each` always runs to completion (no droppable iterator), so lazy advancement solves a problem that can't happen. |
| `for_each_raw` tick advancement | None â€” reads tick to filter, doesn't advance | No `&mut World` available. Consistent with deferred-advancement pattern: tick used for filtering but advanced only when `for_each`/`execute` (the `&mut World` paths) runs. |

## WorldQuery Trait Extension

Add `changed_ids` to the `WorldQuery` trait:

```rust
fn changed_ids(registry: &ComponentRegistry) -> FixedBitSet;
```

| Type | Returns |
|---|---|
| `&T` | empty |
| `&mut T` | empty |
| `Entity` | empty |
| `Option<&T>` | empty |
| `Changed<T>` | `{ registry.id::<T>() }` |
| Tuples 1-12 | bitwise union of all terms |

Generated by the existing `impl_world_query_tuple!` macro â€” same expansion pattern as
`required_ids`. Called once during `scan::<Q>()`, result captured by the scan closure.

## Scan Closure Capture

In `QueryPlanner::scan::<Q>()` and `scan_with_estimate::<Q>()`, capture `changed_ids`
alongside `required`:

```rust
let changed = Q::changed_ids(self.components);
```

The `FixedBitSet` is cloned into each compiled closure. Inside the archetype iteration
loop, after the `required.is_subset` check:

```rust
// Fast path: no Changed<T> terms â€” skip entirely
if !changed.is_clear() {
    let dominated = changed.ones().all(|bit| {
        let comp_id = ComponentId(bit);
        arch.column_index(comp_id)
            .map_or(false, |col| arch.columns[col].changed_tick.is_newer_than(tick))
    });
    if !dominated { continue; }
}
```

The `changed.is_clear()` guard means queries without `Changed<T>` pay zero cost.

## Type Signature Changes

All three internal closure type aliases gain a `Tick` parameter:

```rust
// Before
type CompiledForEach    = Box<dyn FnMut(&World, &mut dyn FnMut(Entity))>;
type CompiledForEachRaw = Box<dyn FnMut(&World, &mut dyn FnMut(Entity))>;
type EntityCollector    = Box<dyn FnMut(&World, &mut ScratchBuffer)>;

// After
type CompiledForEach    = Box<dyn FnMut(&World, Tick, &mut dyn FnMut(Entity))>;
type CompiledForEachRaw = Box<dyn FnMut(&World, Tick, &mut dyn FnMut(Entity))>;
type EntityCollector    = Box<dyn FnMut(&World, Tick, &mut ScratchBuffer)>;
```

Callers load `self.last_read_tick` once and pass it through. The `Tick` type is already
`pub(crate)`, so this stays internal.

## Plan-Instance Tick State

`QueryPlanResult` gains one field:

```rust
last_read_tick: AtomicU64,  // initialized to 0
```

Initialized to 0 because `World` ticks start at 1, so the first call sees everything as
"changed" â€” same behavior as `world.query()` starting with `Tick::default()`.

### Advancement Rules

| Method | Reads tick | Advances tick | Mechanism |
|---|---|---|---|
| `for_each(&mut self, &mut World)` | Yes | Yes | `world.next_tick()` stored after scan |
| `for_each_raw(&mut self, &World)` | Yes | No | No `&mut World` |
| `execute(&mut self, &mut World)` | Yes | Yes | Same as `for_each` |

## Join Behavior

Each `scan::<Q>()` call captures its own `changed_ids` from its own `Q`. In a join plan:

```rust
planner.scan::<(Changed<Pos>, &Vel)>()     // left: changed_ids = {Pos}
    .join::<(&Team,)>(JoinKind::Inner)      // right: changed_ids = {}
    .build()
```

- Left collector filters archetypes by `Changed<Pos>`
- Right collector has empty `changed_ids` â€” no change filter
- All collectors share the plan's single `last_read_tick`
- Join intersection happens after both sides are collected and filtered

## What Changes

- **`query/fetch.rs`**: Add `changed_ids` to `WorldQuery` trait + all impls + tuple macro
- **`planner.rs`**: Capture `changed_ids` in scan closures, add `Tick` parameter to
  closure types, add `last_read_tick: AtomicU64` to `QueryPlanResult`, advance tick in
  `for_each`/`execute`

## What Doesn't Change

- `world.query()` path â€” untouched
- `QueryWriter` / `DynamicCtx` tick handling â€” independent mechanism
- Public API signatures of `for_each`, `for_each_raw`, `execute` â€” unchanged
- Predicate filters â€” orthogonal, still fused the same way
- `SubscriptionBuilder` â€” independent from execution-time change detection

## Invariants

1. **Zero-cost when unused**: `changed.is_clear()` fast path means queries without
   `Changed<T>` do not touch column metadata or compare ticks.
2. **First call sees everything**: `last_read_tick` starts at 0, all column ticks are >= 1.
3. **Immediate advancement**: `for_each`/`execute` advance the tick synchronously after
   scan completes. No lazy flag needed â€” these methods always run to completion.
4. **Read-only path defers**: `for_each_raw` uses the tick for filtering but cannot
   advance it (no `&mut World`). Tick advances on next `for_each`/`execute` call.
5. **Per-plan independence**: each `QueryPlanResult` owns its own tick state. Two plans
   over the same query type track changes independently.
